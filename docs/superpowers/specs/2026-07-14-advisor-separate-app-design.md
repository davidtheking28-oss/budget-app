# Advisor Platform as a Separate App — Design

## Purpose

The advisor desktop experience was originally built as CSS that reshapes
`index.html`'s mobile layout into a sidebar on wide viewports
(`body.is-advisor` + `@media(min-width:1024px)`, see
`docs/superpowers/specs/2026-07-13-advisor-desktop-platform-design.md`). Seeing
it live, the user found it indistinguishable from the regular mobile budget
view — not the distinct "advisor platform" feel intended. This spec replaces
that approach: the advisor experience becomes its own application, sharing
the same Supabase backend and data as the client-facing budget app but with
its own codebase, build, and desktop-first design language.

This supersedes the "Code organization" section of the prior spec (single
file, logically separated) — see memory `advisor-code-single-file` (updated
2026-07-14). The data model and RLS work already built (`advisor_clients`
table, `budget_data` RLS extension, invite/redeem flow) are unaffected and
are reused as-is.

## Non-goals (V1)

- CRM (per-client notes/tasks/meetings), AI insights, advisor-facing push
  notifications — unchanged from the prior spec, still deferred to later
  steps.
- Native app — this stays a responsive web app.
- Changing the invite/redeem flow itself (client generates code in
  `index.html` Settings, advisor redeems it) — that flow is done and stays
  in `index.html`, since it's the client's action, not the advisor's.

## Architecture

A new React + Vite application, source at `advisor-app/`, built and deployed
to the `/advisor/` sub-path of the same GitHub Pages site
(`https://davidtheking28-oss.github.io/budget-app/advisor/`). `index.html`
stays a single vanilla file exactly as it is today, minus the desktop-advisor
CSS/JS that is being removed (see "Cleanup" below) — it keeps its own
Supabase Auth session, its own mobile-first layout, and the client-side
invite-generation/redeem UI in Settings (both directions: a client makes a
code, and — since redeeming is also something a client-turned-advisor does
from their own account today — that UI stays put; nothing about the
invite/redeem mechanism moves).

Both apps talk to the same Supabase project (`fnklrqxwyeibfptaxewf`), the
same `budget_data` table, and the same `advisor_clients` table. No new
backend work is introduced by this spec — Task 1/2/7's tables, RLS, and RPC
(`claim_advisor_invite`) are consumed as-is.

## Repo structure & build

```
budget-app/
  index.html              # unchanged in structure; desktop-advisor code removed
  advisor-app/             # new: React + Vite source
    src/
    package.json
    vite.config.js         # base: '/budget-app/advisor/', build.outDir: '../advisor'
  advisor/                 # build output — git-ignored, generated in CI
```

`vite.config.js` sets `base: '/budget-app/advisor/'` so built asset URLs
resolve correctly under the GitHub Pages sub-path, and `build.outDir:
'../advisor'` so the build lands at the repo root as `advisor/`, alongside
`index.html`, ready for the existing static-file Pages deploy to pick up.

`advisor/` is added to `.gitignore` — it's a build artifact, regenerated by
CI on every deploy, never committed.

### CI changes (`.github/workflows/deploy.yml`)

A build step is added before the existing artifact upload:

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: 20
- name: Build advisor app
  working-directory: advisor-app
  env:
    VITE_SUPABASE_ANON: ${{ secrets.SUPABASE_ANON }}
  run: |
    npm ci
    npm run build
```

placed after the existing `Inject Supabase anon key` step (which continues
to handle `index.html` exactly as today via string replace) and before
`actions/upload-pages-artifact@v3`, so the artifact upload (which already
uploads `.` — the whole repo root) picks up the freshly generated `advisor/`
directory alongside the untouched `index.html`. The advisor app reads its
Supabase anon key from `import.meta.env.VITE_SUPABASE_ANON` at build time
(Vite's standard env-var convention), rather than the string-replace
placeholder trick `index.html` uses — no `__SUPABASE_ANON__` placeholder
needed in the React source.

## Screens (V1)

1. **Login** — Supabase Auth email magic-link (`signInWithOtp`), matching
   `index.html`'s existing auth method exactly (same Supabase project, same
   mechanism — no password login is introduced), its own desktop-first
   screen, its own visual design (see "Design language" below — deliberately
   not navy/indigo).
2. **Client list** — reads `advisor_clients` where `advisor_id = auth.uid()
   and status = 'active'`, one row per client (name/email), click-through
   into that client's budget. Directly replaces `renderAdvisorClients()` /
   `#page-advisorClients` from `index.html` (which is being removed).
3. **Client budget management** — a full new desktop-first UI (not a port or
   clone of `index.html`'s mobile screens) covering: dashboard/overview,
   expenses, budget/categories, analysis — for whichever client is currently
   selected. Reads/writes `budget_data` scoped to the client's `user_id`,
   protected by the existing advisor RLS branch on that table. No new
   client-switching mechanism is needed server-side (RLS already scopes
   correctly per-request); client-side, the React app just tracks "which
   client am I currently viewing" in its own state and re-queries.

## Design language

Independent from `index.html`'s client-facing navy/indigo theme
(`--bg:#0d1526`, `--accent:#4f83ff`) — this app gets its own palette,
typography, and component style, established during implementation (not
fixed in this spec) so it reads as a distinct professional tool rather than
a reskinned version of the client app. `docs/superpowers/specs/` and the
`frontend-design`/`ui-ux-pro-max` skills are available if a more structured
design pass is wanted once implementation starts.

## Cleanup in `index.html`

Remove (no longer needed — replaced by the separate app):

- `@media(min-width:1024px)` sidebar CSS block gated on `body.is-advisor`
  (index.html ~line 100-111).
- `document.body.classList.toggle('is-advisor', ...)` and the
  nav-visibility toggle for `#nav-advisorClients` in `_applyAdvisorGate()`
  (keep `_isAdvisor`/`_loadAdvisorStatus()` themselves — still meaningful:
  index.html could later link out to `/advisor/` only for confirmed
  advisors, though that's not required for this spec).
- `#page-advisorClients`, `#nav-advisorClients` markup, `renderAdvisorClients()`.
- `enterClientBudget()` / `exitClientBudget()`, `#advisorViewBar` markup/CSS,
  `_advisorViewingClient` / `_advisorPriorHouseholdOwner` globals.

Keep (still needed — these belong to the client's own account, not the
advisor console):

- `createAdvisorInvite()` / `renderAdvisorLinkSection()` / `#advisorLinkSection`
  (client generates a code for their advisor).
- `redeemAdvisorInvite()` / `#advisorRedeemCodeInput` (anyone, including a
  would-be advisor, can redeem a code from their own Settings — this is the
  one piece of advisor-facing UX that intentionally stays in `index.html`,
  since it's an action any authenticated user takes from their own account
  before they have an advisor console to go to).
